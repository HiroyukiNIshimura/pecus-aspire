# 📝 生成AI活用：人間向けテスト仕様書テンプレート

人間が実施主体となるテストについて、生成AIに**たたき台作成・観点整理・ケース分解・証跡整理**を支援させるための、共通テンプレートを定義します。

Ver1.0.0

---

## このテンプレートの目的

このテンプレートは、以下のような**人間が最終実施・最終判断するテスト**の仕様書を作るために使います。

- ST（System Test）
- 受け入れテスト
- UAT
- 目視確認を含む業務シナリオテスト
- リリース前確認
- 本番同等環境での運用確認

> **生成AIは仕様書の草案作成と整理を支援できるが、合否判定そのものを代行するものではない。**

---

## 第1章：基本情報

| 項目 | 内容 |
|---|---|
| テスト名称 | 例: リリース前 ST（2026-06 Sprint 5） |
| テスト目的 | 何を確認するテストか |
| 対象システム | 例: Frontend / WebApi / BackFire / Redis / PostgreSQL |
| 対象環境 | 例: 準本番 / 本番同等 / 検証環境 |
| テスト実施者 | 人間の担当者名またはロール |
| テスト支援AI | 例: 観点整理 / ケース草案 / 証跡整理 / 障害票初稿 |
| 実施期間 | YYYY-MM-DD ～ YYYY-MM-DD |
| 関連ドキュメント | `4_IT.md`, `5_UIテスト.md`, `6_ST.md` など |

---

## 第2章：テストの目的とゴール

### 1. テスト目的

- 本テストで確認したいことを箇条書きで記載
- 例:
  - 本番同等環境における主要機能の安定性確認
  - 高負荷時の API 応答劣化有無の確認
  - 権限制御、監視、障害時の切り分け性確認

### 2. ゴール

- このテストで「完了」とみなす条件を記載
- 例:
  - Critical / High の未解決障害が残っていない
  - Go/No-Go 判断に必要な証跡が揃っている
  - 保留課題に対する受容条件が明文化されている

### 3. 非ゴール

- このテストでは確認しない範囲を明示
- 例:
  - 単体ロジックの詳細確認
  - UI 微調整のみを目的とした回帰確認
  - 本番承認そのもの

## 第3章：テスト対象範囲

### 対象機能

| ID | 機能 / 観点 | 対象理由 | 重要度 |
|---|---|---|---|
| TGT-001 | 例: ログイン〜主要ダッシュボード導線 | 利用頻度が高い | High |
| TGT-002 | 例: 高負荷時のタスク一覧 API | SLA 影響あり | High |

### 対象外機能

| 項目 | 理由 |
|---|---|
| 例: 管理者専用バッチの詳細性能 | 今回のテストスコープ外 |

## 第4章：前提条件

### 環境前提

- 対象環境名
- デプロイ済みバージョン
- 利用するデータセット
- 監視 / ログ / ダッシュボード利用可否
- 外部連携の状態

### データ前提

- 利用ユーザー
- 事前投入データ
- 検証に必要な権限状態
- 後処理の要否

### 実行前チェック

- [ ] 対象バージョンを確認した
- [ ] 対象環境の URL / 接続先を確認した
- [ ] テストデータの初期状態を確認した
- [ ] ログ / メトリクス / 証跡取得先を確認した

## 第5章：テスト観点一覧

| 観点ID | 観点 | 観点の意図 | 実施方法 | 期待結果 | 証跡 |
|---|---|---|---|---|---|
| VIEW-001 | 例: 主要画面表示 | 初期導線が成立すること | 人間が画面遷移を実施 | 正常表示される | 画面キャプチャ |
| PERF-001 | 例: 高負荷時応答 | 応答劣化が許容範囲か | 負荷投入 + 結果確認 | 閾値内で応答 | メトリクス |
| SEC-001 | 例: 権限制御 | 権限外アクセス不可 | ロール切替で検証 | 403 / 非表示 | ログ / 画面証跡 |

> **観点の列挙は生成AIに草案作成させやすく、人間レビューで不足や過剰を調整しやすい。**

## 第6章：テストケース詳細

以下を 1 ケースごとに繰り返す。

### ケース: TC-001 例）一般ユーザーで主要ダッシュボードを表示できる

| 項目 | 内容 |
|---|---|
| ケースID | TC-001 |
| 観点ID | VIEW-001 |
| 目的 | 一般ユーザーの主要導線が成立することを確認 |
| 前提条件 | 一般ユーザーでログイン可能 |
| 操作手順 | 1. ログインする 2. ダッシュボードを開く |
| 期待結果 | 初期表示に必要な情報が欠落なく表示される |
| 実施者確認ポイント | 文言違和感、表示崩れ、導線の分かりやすさ |
| 取得証跡 | スクリーンショット、ログ、メトリクス |
| 重要度 | High |
| 結果 | 未実施 / Pass / Fail / 保留 |
| 備考 | 既知制約があれば記載 |

## 第7章：人間が必ず確認するポイント

以下は、生成AIに草案を作らせても**最終的には人間が確認すべき項目**です。

### 1. 体験品質

- 文言の自然さ
- 見た目の違和感
- 導線の分かりやすさ
- 誤解を招く UI 表現の有無

### 2. 業務妥当性

- 業務フローとして破綻していないか
- 現場運用で受け入れられるか
- 例外系で誤操作を誘発しないか

### 3. 非機能の最終判断

- 性能値が許容範囲か
- セキュリティ結果が受容可能か
- 監視 / アラート体制で運用可能か
- 残課題を抱えたままリリースできるか

---

## 第8章：証跡管理

| 証跡種別 | 用途 | 保存先 | 備考 |
|---|---|---|---|
| 画面キャプチャ | 見た目確認 | 例: `/evidence/screenshots/` | ケースID紐付け |
| ログ | 異常分析 | 例: ログ基盤 / ファイル保存先 | 時刻・対象サービス必須 |
| メトリクス | 性能分析 | 例: Grafana / APM | 対象期間必須 |
| 動画 / trace | 再現支援 | 例: Playwright artifacts | UI系で利用 |
| スキャン結果 | セキュリティ確認 | 例: scan-results/ | ツール名と実行時刻を記録 |

### 証跡記録ルール

- ケースIDを必ず付与する
- 取得時刻を残す
- 「何をもって Pass / Fail としたか」が追えるようにする
- 生成AIが要約した場合でも、生の保存先を残す

---

## 第9章：障害票 / リスク票記載ルール

### 障害票に最低限含める内容

- ケースID
- 発生条件
- 再現手順
- 実際の結果
- 期待結果
- 影響範囲
- 証跡参照先
- 一次切り分け結果（AI初稿可）

### リスク票に最低限含める内容

- リスク内容
- 想定影響
- 発生条件
- 回避策 / 暫定対応
- リリース判断への影響
- 受容判断者

---

## 第10章：進捗管理

| 指標 | 値 |
|---|---|
| 総ケース数 | N |
| 未実施 | n |
| Pass | n |
| Fail | n |
| 保留 | n |
| 進捗率 | (Pass + Fail + 保留) / N |

---

## 第11章：完了判定（DoD）

- テスト対象範囲と対象外が明確化されている
- 重要観点のケースが実施済みである
- Fail / 保留に対する障害票またはリスク票が整備されている
- 主要証跡が参照可能な状態で保存されている
- 人間が最終確認すべきポイントが確認済みである
- Go/No-Go 判断材料が揃っている

---

## 第12章：生成AIへの依頼テンプレート

以下のような依頼で、生成AIに人間向けテスト仕様書の草案を作らせやすい。

```markdown
### 依頼例

以下を前提に、人間が実施するためのテスト仕様書草案を作成してください。

- 対象フェーズ: ST / UAT / 受け入れテスト
- 対象機能: [ここに記載]
- 対象環境: [ここに記載]
- 重要観点: [性能 / セキュリティ / UI / 業務導線 など]
- 既知の残課題: [ここに記載]
- 期待する成果物:
  - テスト観点一覧
  - テストケース一覧
  - 人間が目視確認すべきポイント
  - 必要証跡一覧
  - 障害票に含めるべき項目

制約:
- 人間が最終実施・最終判定する前提とする
- 推測で業務ルールを追加しない
- 不明点は TODO として明記する
```

## まとめ

このテンプレートを使うことで、生成AIは

- **観点のたたき台作成**
- **ケース分解**
- **証跡整理項目の明確化**
- **障害票 / リスク票初稿の標準化**

を支援できるようになる。

一方で、**人間が実施・確認・判定すべき領域をあえて残す**ことで、テスト仕様書としての現実性と安全性を両立できる。
