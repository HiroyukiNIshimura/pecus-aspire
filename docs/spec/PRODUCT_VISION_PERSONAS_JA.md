# Coati（Pecus Aspire） — プロダクトビジョンとユーザーペルソナ

最終更新: 2025-12-12

このドキュメントは `pecus-aspire` プロジェクト（Coati）のプロダクトビジョン、ミッション、主要なユーザーペルソナ、代表的なユーザージャーニー、成功指標（KPI）をまとめたものです。開発、プロダクト設計、ドキュメント作成、営業/マーケティングの共通基盤として利用してください。

## 1. プロダクトビジョン（Vision）
Coati は「管理をなくし、信頼でつながる。場所も時間も超えて、チームが自律的に動くための『場』」です。

- 長期目標: 「報告・連絡・相談」を不要にする。すべての活動を透明にすることで、誰もが自律的に判断し、本来の価値創造に没頭できる世界を作ります。
- コア価値: 境界のないコラボレーション、性善説に基づく透明性、非同期でもスムーズな連携、自由なワークスタイル。

## 2. ミッション（Mission）
チームの「自律神経」になる。
いちいち聞かなくても「今なにが起きているか」が空気のように分かる状態を作り、マイクロマネジメントを撲滅する。

- シンプルさ: ツールを使っていることを意識させない。
- 信頼性: 「監視」ではなく「透明性」によって、メンバー間の信頼を担保する。
- 拡張性: 組織の形が変わっても、柔軟にフィットし続ける。

## 3. ターゲットユーザー像（サマリ）
- 場の設計者（Admin）
- チームの交通整理役（Leader/PM）
- 価値を生み出す主役（Contributor）
- システムの守護者（SRE）
- 応援者・観客（Stakeholder）

以下、典型的なペルソナを詳細に定義します。

## 4. ユーザーペルソナ（詳細）

### Persona A: 場の設計者（Admin）
- 概要: 自由で安全なコラボレーションの場（ワークスペース）を整える人。
- 役職例: ITマネージャー、コミュニティマネージャー
- 目的:
  - メンバーが迷わず動ける「場」の提供
  - 最低限のルール設定で、最大限の自由を保証する
  - チームの健全性を保つ
- 主な要求:
  - 誰でも簡単に招待・参加できる仕組み
  - 「やってはいけないこと」だけを定義するシンプルな権限設定
  - チームの代謝（参加・脱退）をスムーズにする
- フラストレーション:
  - 煩雑な承認フローや設定作業に時間を取られること
  - メンバーを縛り付けるような設定を強いられること
- 優先機能:
  1. 役割に応じたシンプルな権限設定 (RBAC)
  2. 組織/ワークスペースの柔軟な作成・変更
  3. 活動ログによる健全性の確認
- 成功指標 (KPI): メンバーが活動開始できるまでの速さ、管理作業の少なさ

---

### Persona B: チームの交通整理役（Leader/PM）
- 概要: 命令するのではなく、チームの障害を取り除き、流れを良くする人。
- 役職例: チームリーダー、スクラムマスター
- 目的:
  - チームのボトルネック（詰まっているところ）の発見と解消
  - メンバーが一番重要なことに集中できる環境作り
  - タスクの依存関係を整理し、手戻りを防ぐ
- 主な要求:
  - 「今、誰が何で止まっているか」が一目でわかるビュー
  - 優先順位の明確化と共有
- フラストレーション:
  - 状況確認のための会議やチャットに時間を取られること
  - 情報共有漏れによる無駄な作業
- 優先機能:
  1. 状況を多角的に見渡せる高度なビュー
  2. 依存関係の可視化と整理
- 成功指標 (KPI): チームのフロー効率、会議時間の削減率

---

### Persona C: 価値を生み出す主役（Contributor）
- 概要: 自律的にタスクを選び、実行し、価値を届ける人。
- 役職例: エンジニア、デザイナー、営業、サポート
- 目的:
  - 自分のリズムで、最高のパフォーマンスを発揮する
  - チームの文脈を理解し、自ら判断して動く
- 主な要求:
  - 「今やるべきこと」への最速アクセス
  - 過去の経緯や文脈への透明なアクセス（情報の非対称性をなくす）
  - 自分の貢献やスキルの可視化
- フラストレーション:
  - 必要な情報を探すのに時間がかかること
  - 自分の作業が誰かの作業とぶつかって無駄になること
- 優先機能:
  1. 自分専用のコックピット（ダッシュボード）
  2. 非同期でも伝わるコメント・通知機能
  3. 文脈を理解するための添付・プレビュー機能
- 成功指標 (KPI): 没頭できる時間（フロー状態）の確保、自律的なタスク消化率

---

### Persona D: システムの守護者（SRE）
- 概要: サービスという「インフラ」を止めず、進化させ続ける人。
- 役職例: SRE、インフラエンジニア
- 目的:
  - サービスの健全性を保ち、ユーザーの体験を損なわない
  - 開発者が安心して変更を加えられる基盤を作る
- 主な要求:
  - システムの「健康状態」のリアルタイムな可視化
  - 何かあった時の「自己修復」や「迅速な復旧」の手順
- フラストレーション:
  - 予期せぬエラーで夜中に起こされること
  - 手作業での復旧作業
- 優先機能:
  1. 異常の早期検知と自動通知
  2. 安全なデータベース更新・復旧の仕組み
  3. 詳細なログによる原因究明の容易化
- 成功指標 (KPI): サービスの稼働率、復旧までのスピード、夜間呼び出しの少なさ

---

### Persona E: 応援者・観客（Stakeholder）
- 概要: チームの外からプロジェクトを見守り、支援する人。
- 目的:
  - プロジェクトの健全性を確認し、必要な支援を行う
  - チームの邪魔をせずに、状況を把握する
- 主な要求:
  - ひと目でわかる「今の状態」（サマリ）
- フラストレーション:
  - 状況を知るためにわざわざ会議を開くこと
- 優先機能:
  1. 経営・外部向けダッシュボード
- 成功指標 (KPI): 状況確認のための会議ゼロ

## 5. 代表的なユーザージャーニー

### 新しいチームを立ち上げる（Admin → TeamLead → Users）
1. Admin が新規ワークスペースを作成。
2. Admin が初期ロールと数名のユーザーを招待・登録。
3. TeamLead が初期タスク群を作成し、各担当にアサイン。
4. Users は通知を受け取り、自身のダッシュボードでタスクに着手。
5. SRE はマイグレーションやジョブの監視を行い、環境安定を確認。

期待される価値: 立ち上げ時間の短縮、初期ミスの低減、早期の生産性発揮。

---

### 変更の同時発生（競合）を扱う流れ
1. User A がタスクの編集を開始すると、User B の画面には「User A が編集中」と表示され、編集ボタンが押せなくなる（無駄な衝突の予防）。
2. User A が編集を終えると、User B はすぐに編集可能になる。
3. 万が一、タイミングのズレなどで同時に保存された場合は、後から保存したユーザーに「競合」を通知し、上書きするかどうかを判断してもらう（最終防衛）。

期待される価値: 「せっかく書いたのに保存できなかった」という悲劇を未然に防ぐ。お互いの気配を感じながら、スムーズに連携できる。

## 7. プライバシー・セキュリティの運用上の考慮点
- フロントは sessionId のみ Cookie に保存（httpOnly）し、access/refresh token は Redis 管理—XSSリスクの低減。
- アクティビティ（Activities）はチームの振り返りや分析のために保持する（厳密なセキュリティ監査ログではないため、IPやUserAgent等は収集しない）。
- 権限管理は厳格に（最小権限の原則）—デフォルトロールは限定的に設定。
- データ保持ポリシーとエクスポート/削除フローを仕様化する（GDPR 等の準拠を想定）。

## 8. プライオリティとローンチ M.V.P.（提案）
MVP の優先機能セット（最小限で価値を出すため）:
1. ユーザー/組織/ワークスペースの作成と管理（Admin ワークフロー）
2. タスクの作成/編集/アサイン（TeamLead & Contributor）
3. アクティビティログ（履歴）と基本的な検索/フィルタ
4. 認証（sessionId + Redis トークン管理）と RBAC の基本
5. Hangfire を使ったバックグラウンドジョブ（メール・バッチ）と監視
